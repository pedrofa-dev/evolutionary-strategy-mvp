🧪 How to Design Experiments

Designing good experiments is critical.
A poorly designed run will produce misleading or useless results.

🎯 Goals of an Experiment

Every experiment should answer a clear question:

Which configuration is more robust?
Does increasing mutation improve exploration?
Do certain datasets produce better generalization?
Is a strategy overfitting?

If you don’t have a question → the experiment is probably noise.

🧱 Core Dimensions to Control
1. Configuration (Genetic Setup)

This defines how the system explores the search space.

Examples:

mutation rates
population size
survivor count
mutation profile

👉 Use multiple configs to compare behaviors:

balanced
exploratory
conservative
2. Datasets

Datasets define the environment where agents are evaluated.

Important aspects:

market (BTC, ETH, etc.)
timeframe (1m, 1h, 1d)
volatility regimes
trending vs ranging periods

👉 Diversity is key for generalization.

3. Seeds (Stochastic Stability)

Evolution is stochastic.

👉 Always use multiseed to validate:

1 seed → unreliable
3 seeds → basic signal
5+ seeds → strong validation
4. Presets (Run Intensity)

Presets define how “serious” the experiment is.

screening → fast, exploratory
full → slower, more reliable
🧪 Types of Experiments
🔹 1. Screening Experiments

Goal:

compare configs quickly

Setup:

preset=screening
multiple configs
small datasets

Use when:

tuning parameters
exploring ideas
🔹 2. Validation Experiments

Goal:

confirm robustness

Setup:

preset=full
multiseed
diverse datasets

Use when:

something looks promising
deciding if a strategy is usable
🔹 3. Specialization Experiments

Goal:

optimize for a specific market

Setup:

narrow dataset (e.g. BTC only)
tuned configs

Use when:

you want performance, not generalization
⚠️ Common Mistakes
❌ 1. Trusting single runs

A single run can look great and be completely random.

❌ 2. Using too few datasets

Leads to:

overfitting
fragile strategies
❌ 3. Over-optimizing configs early

Don’t fine-tune before you understand behavior.

❌ 4. Ignoring validation metrics

Train performance means nothing alone.

🧠 Best Practices
✔ Compare, don’t optimize blindly

Always run:

multiple configs
same datasets
same seeds
✔ Look for consistency

Good configs:

produce similar results across seeds
don’t collapse on validation
✔ Keep experiments simple

Change one thing at a time.

✔ Use the system as intended
evolution → exploration
champions → filtering
reporting → insight
